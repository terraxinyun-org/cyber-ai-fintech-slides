Module 01

Threat Landscape

20 minutes

The CIA Triad

The three pillars of information security — applied to VoltPay:

🔒 Confidentiality

Only authorized people can access data.

VoltPay: Merchant transaction data, customer PII, API keys must stay private.

🛡️ Integrity

Data hasn't been tampered with.

VoltPay: Payment amounts, merchant records, audit logs must be accurate.

⚡ Availability

Systems work when needed.

VoltPay: Payment processing must handle SGD 180M/month without downtime.

CIA triad is the foundation. Every security decision maps back to these three. Use VoltPay examples to make it concrete. Ask students: "Which is most important for a payment company?"

4 Common Attacks

🎣 Phishing

Fake emails/messages that trick users into revealing credentials. 91% of cyberattacks start with phishing.

VoltPay risk: CFO receives fake "urgent wire transfer" email from "CEO".

🔐 Ransomware

Malware encrypts all files, demands payment (crypto) for decryption key.

VoltPay risk: Payment DB encrypted → 2,400 merchants can't process transactions.

🌊 DDoS

Overwhelm servers with traffic so legitimate users can't connect.

VoltPay risk: Payment gateway down during peak hours → merchant revenue loss.

🎭 Social Engineering

Manipulating people to bypass security. Exploits trust, urgency, authority.

VoltPay risk: Attacker calls helpdesk pretending to be Kevin (IT Manager).

Keep it concrete with VoltPay examples. Ask: "Which of these can technology alone prevent?" (None — humans are always the weakest link.)

AI in Cybersecurity

It's a two-sided arms race:

⚔️ Attackers Use AI

  • AI-generated phishing emails (no more typos)
  • Deepfake voice calls ("Hi, this is your CEO...")
  • Automated vulnerability scanning at scale
  • AI-written malware that evades detection

🛡️ Defenders Use AI

  • Anomaly detection in network traffic
  • Log analysis across millions of events
  • Automated incident response playbooks
  • AI coding assistants (like today's lab!)

Today you'll use OpenCode — a free, open-source AI agent — as your security co-pilot.

This is the "why AI" slide. Students should understand AI amplifies both sides. Key message: AI is a tool, not a replacement. You still need to think critically.

Exercise 1: Port Scanning

What is nmap?

Network mapper — discovers which "doors" (ports) are open on a system.

Think of ports like apartment doors in an HDB block. Each one leads to a different service.

nmap localhost

What to Look For

  • Port 5000 — Flask/Werkzeug (our target!)
  • Port 5601 — Kibana (SIEM)
  • Port 9200 — Elasticsearch
  • Other open ports = other services

Expected Output

PORT     STATE SERVICE
5000/tcp open  upnp
5601/tcp open  esmagent
7681/tcp open  unknown
8080/tcp open  http-proxy
8081/tcp open  blackice-icecap
9200/tcp open  wap-wsp

🏆 Achievement: Reconnaissance Complete

Walk through nmap output. Explain what each port means. Key teaching moment: Reconnaissance is legal and common — it's scanning, not attacking. Real pen testers start every engagement this way.

Exercise 2: Vulnerability Scanning

What is nikto?

Automated web scanner — checks for misconfigurations and known vulnerabilities.

nikto -h http://localhost:5000

Key Findings

  • MISSING X-Content-Type-Options
  • MISSING X-XSS-Protection
  • MISSING X-Frame-Options

Maps to OWASP A05:2021 — Security Misconfiguration

Why Headers Matter

X-Frame-OptionsPrevents clickjacking
X-XSS-ProtectionBrowser XSS filter
X-Content-TypePrevents MIME sniffing

🏆 Achievement: Inspector General

nikto is noisy — in real pen tests, it would be detected by a WAF. These missing headers are "low-hanging fruit" that every security audit catches.

Exercise 3: SQL Injection

OWASP A03:2021 — Injection

Unsanitized user input inserted directly into database queries.

The Vulnerability

Merchant search endpoint passes input directly to SQL:

SELECT * FROM merchants
WHERE name LIKE '%user_input%'

The Exploit

curl "http://localhost:5000/api/merchants\
/search?q=' OR '1'='1"

Result: Returns ALL merchants — including hidden internal records.

Impact on VoltPay

  • Database extraction — All merchant data exposed
  • Authentication bypass — Login without valid credentials
  • Data modification — Alter payment amounts
  • PDPA violation — 2,400 merchants' data

🏆 Achievement: SQL Sorcerer
🔍 Hidden flag: Transaction 521 contains confidential data

This is the most impactful vulnerability. Walk through the SQL query step by step. Show what ' OR '1'='1 does to the query structure. Key: the database can't tell the difference between code and data. Ask: "How would you fix this?" (Parameterized queries / prepared statements)

Exercise 4: Cross-Site Scripting (XSS)

What is XSS?

Attacker injects JavaScript that runs in other users' browsers.

<script>alert('XSS')</script>

The Exploit

Search parameter reflected without escaping:

/dashboard?search=
<script>alert(document.cookie)</script>

Real-World Impact

  • Steal session cookies → account takeover
  • Redirect to phishing page
  • Modify displayed transaction amounts

OWASP A03:2021 — Injection

Same category as SQLi — different injection point (HTML vs SQL).

VoltPay Scenario

Attacker sends merchant a link:

voltpay.sg/dashboard?search=<script>steal()</script>

Merchant clicks → session stolen → attacker accesses their payment data.

🏆 Achievement: Script Kiddie No More

Demonstrate in browser if possible. Key insight: API endpoints return JSON (safe), HTML pages are vulnerable. Ask: "Why does the API not have this problem?" (JSON is data, HTML is code)

Exercise 5: IDOR

OWASP A01:2021 — Broken Access Control

#1 on OWASP Top 10. Insecure Direct Object Reference — no authorization check on resource access.

The Vulnerability

Transaction API uses sequential IDs with no authorization check:

# Logged in as Sarah (merchant_id=2)
# But can access ANY transaction:
curl /api/transactions/1    # ✅ Not hers!
curl /api/transactions/521  # ✅ Confidential!

Business Impact

  • Competitor intelligence — see other merchants' revenue
  • Customer data exposure — all transaction details
  • Regulatory violation — unauthorized data access

🏆 Achievement: Access Denied... Not

IDOR is the #1 vulnerability class. Very common in APIs. The fix is simple: check if the logged-in user owns the requested resource. Ask: "Would encryption fix this?" (No — it's an authorization problem, not a confidentiality one.)

Exercise 6: Default Credentials

The Vulnerability

Admin panel accessible with factory-default password:

admin / admin123

OWASP A07:2021 — Identification and Authentication Failures

Admin Panel Access

  • All user accounts and roles
  • All merchant data
  • API keys and secrets
  • Full audit logs

Game Over

With admin access, an attacker can:

  • Export entire database
  • Create backdoor accounts
  • Modify transaction records
  • Disable security logging

This is a full system compromise.

🏆 Achievement: Master Key

This is embarrassingly common. Real-world examples: Mirai botnet exploited default credentials on IoT devices. Key point: Kevin reveals in Module 04 that admin/admin123 is known to ALL developers.

What is a SIEM?

Security Information and Event Management — the command center of any Security Operations Center (SOC).

Collects logs from every system, normalizes them, and enables search + correlation.

Our SIEM Stack

  • Elasticsearch — stores & searches logs
  • Kibana — visualizes & explores data
  • 10K+ log entries across 5 indices
  • Pre-loaded attack scenario embedded in noise
Real SOCs use Splunk, Microsoft Sentinel, or ELK (Elasticsearch/Logstash/Kibana). We're using the same ELK stack that many companies use in production.

Log Sources

IndexSourceContains
logs-authAuthenticationLogin successes/failures, usernames, source IPs
logs-firewallNetworkConnection logs, source/dest IPs, ports, actions
logs-dnsDNSDomain queries, query types, response codes
logs-sysmonProcessProcess creation, command lines, parent processes
logs-webHTTPWeb requests, status codes, user agents, paths

Scenario: VoltPay's SIEM captured logs from a single day. Somewhere in these 10,000 entries is evidence of a brute force attack. Your job: find it.

This mirrors real SOC work. Analysts sift through thousands of events looking for the signal in the noise. The brute force attack is the primary scenario — there are also DNS tunneling and ransomware traces for advanced students.

Investigation: Brute Force Attack

Students query Elasticsearch step by step, guided by AI:

Step 1
Overview
Count auth successes vs failures → huge ratio of failures = something is wrong
Step 2
Find Top Offenders
Aggregate by source IP → one IP has hundreds of failed attempts
Step 3
Identify Targets
Which usernames were targeted? Credential stuffing or targeted attack?
Step 4
Check for Success
Did the attacker eventually get in? Search for successful login from attacker IP
Step 5
Cross-Reference
Check firewall/DNS logs for the same IP → what else did they do post-compromise?
Step 6
AI Synthesis
Build attack timeline, extract IOCs, map to MITRE ATT&CK T1110
This is the most analytical module. Guide students through each query. Key teaching moment: the value of CORRELATION — looking at the same event from multiple data sources. MITRE ATT&CK T1110 = Brute Force technique.

Incident Response Report

Students draft a professional IR report with AI assistance:

Report Sections

  1. Executive Summary — 3 sentences for the CEO
  2. Timeline — chronological attack events
  3. IOCs — attacker IP, usernames, timestamps
  4. Root Cause — why it happened
  5. Impact Assessment — what was compromised
  6. Recommendations — immediate + long-term
  7. Lessons Learned — prevent recurrence

Key IOCs to Extract

  • Attacker IP address(es)
  • Targeted usernames
  • Successful compromise timestamp
  • Post-compromise activities
  • MITRE ATT&CK: T1110

🏆 Achievements: Digital Detective + Report Writer

The IR report template is provided. AI helps structure and fill in the content. Key teaching: writing for the audience — the Executive Summary is for the CEO, the IOCs are for the SOC team. This is a real skill used by incident responders every day.

Governance & Compliance

25 minutes • Stakeholder Interview

What is IT Governance?

Think of it like building codes for technology — just as HDB has safety requirements for buildings, MAS has security requirements for payment systems.

MAS TRM

Technology Risk Management guidelines for financial institutions.

Key sections: Board governance (§3), Secure development (§6), Access control (§9), Data security (§11), Cyber ops (§12), Vulnerability mgmt (§13)

PDPA

Personal Data Protection Act — consent, purpose limitation.

Breach notification within 3 days. Fines up to SGD 1M or 10% annual turnover.

CSA Cyber Trust

Voluntary certification mark from Cyber Security Agency.

Required by DBS for the SGD 5-8M/year partnership.

Governance isn't sexy but it's essential. Without compliance, VoltPay can't operate. Singapore's regulatory environment is one of the strictest in APAC — and that's a feature, not a bug.

AI: Strengths & Limitations

What AI Does Well

  • Explains complex concepts in plain language
  • Analyzes tool output quickly
  • Maps findings to frameworks (OWASP, MITRE)
  • Drafts reports and documentation
  • Suggests next investigative steps
  • Simulates personas for practice

Where AI Falls Short

  • Hallucination — can fabricate facts confidently
  • No real-time verification — can't check if systems are actually patched
  • Over-confidence — rarely says "I don't know"
  • Context limits — loses track in long sessions
  • Lacks judgment — can't weigh business trade-offs
  • No accountability — you sign the report, not the AI

AI is a force multiplier, not a replacement for thinking.

Critical slide. Students must understand AI limitations BEFORE they rely on it professionally. Key message: The person using the AI is still responsible for the output. "You sign the report, not the AI" — this is the most important takeaway.
